# MVP 开发路线图

**项目**：Labhaus  
**版本**：v0.1.0 MVP  
**计划周期**：12 周（3 个月）  
**创建日期**：2026-06-18

---

## 总体目标

**MVP 核心目标**：验证"非技术人员能否用工作流平台批量生产 AI 视频"

**成功标准**：
- ✅ 10 个种子用户能独立完成首个视频
- ✅ 制作时间 < 传统方式 50%
- ✅ 任务成功率 > 90%
- ✅ 至少 2 个用户表示愿意付费

## 风险管理

### 高风险项

| 风险 | 影响 | 概率 | 应对策略 | 负责人 |
|------|------|------|---------|--------|
| 生成质量不稳定 | 高 | 高 | 多模型测试，质量检查 | 后端 |
| 前端性能问题 | 中 | 中 | 虚拟滚动，懒加载 | 前端 |
| API 成本过高 | 高 | 中 | 缓存，降级方案 | 后端 |
| 进度延期 | 高 | 中 | 每周 Review，及时调整 | PM |

### 应急预案

- **Plan B**：如果可视化编辑器延期，可先发布 CLI 版本
- **降级方案**：如果生图质量不稳定，可先用固定模板
- **成本控制**：设置 API 调用上限，防止成本失控

## 质量保证

### 测试策略

1. **单元测试**
   - 覆盖率目标：> 80%
   - 工具：pytest（后端），Jest（前端）
   - 频率：每次提交

2. **集成测试**
   - 场景：端到端工作流
   - 工具：pytest + selenium
   - 频率：每日构建

3. **性能测试**
   - 指标：响应时间、并发数
   - 工具：Locust / k6
   - 频率：每周一次

4. **可用性测试**
   - 对象：10 个种子用户
   - 方法：任务完成测试
   - 频率：Phase 2 和 Phase 3 完成后

### Code Review

- **规则**：所有 PR 需要至少 1 人 Review
- **检查项**：代码规范、测试覆盖、性能影响
- **工具**：GitHub PR Review

## 部署计划

### 开发环境

- **本地开发**：Docker Compose
- **测试环境**：Docker Compose + VPS
- **配置**：.env 文件

### 生产环境（MVP 后）

- **云平台**：AWS / 阿里云 / 自建
- **容器化**：Docker + Kubernetes
- **监控**：Prometheus + Grafana
- **日志**：ELK Stack

### 发布流程

1. **本地测试** → 通过
2. **提交 PR** → Code Review
3. **合并 main** → CI 自动测试
4. **部署 staging** → 集成测试
5. **部署 production** → 监控
